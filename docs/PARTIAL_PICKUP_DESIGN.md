# ออกแบบ: บันทึกการเบิกสินค้า / รับที่หน้าร้าน (Partial Pickup)

## ปัญหา

- ลูกค้าสั่งสินค้ามาในระบบ (order → delivery_trip_items) แสดงเป็น **จำนวนเต็ม**
- ในความเป็นจริง ลูกค้าอาจ **มาเบิกสินค้าไปส่วนหนึ่ง** หรือ **มาเอาที่หน้าร้านเองทั้งหมด**
- ระบบไม่มีการระบุเรื่องนี้ → เวลาจัดทริปจะคิดจำนวนที่ต้องส่งผิด (มากกว่าของจริง) และจัดเรียง/โหลดผิด

## เป้าหมาย

1. **บันทึกได้ว่า** รายการนี้ลูกค้าเบิกไปแล้วกี่หน่วย (หรือรับที่หน้าร้านทั้งหมด)
2. **จำนวนที่ใช้จัดทริป/จัดเรียง** = จำนวนที่สั่ง − จำนวนที่เบิกไปแล้ว (= จำนวนที่ต้องส่งจริง)
3. ยังคง **จำนวนที่สั่งเดิม** ไว้ในระบบ (สำหรับใบแจ้งหนี้/ประวัติ) แยกจาก "จำนวนที่ต้องจัดส่ง"

---

## โครงสร้างข้อมูล (ที่เสนอ)

### 1. ตาราง `delivery_trip_items`

เพิ่มคอลัมน์:

| คอลัมน์ | ประเภท | ความหมาย |
|--------|--------|----------|
| `quantity` | (เดิม) | จำนวนที่ลูกค้าสั่ง (จาก order) — **ไม่เปลี่ยน** |
| `quantity_picked_up_at_store` | numeric DEFAULT 0 | จำนวนที่ลูกค้าเบิก/รับไปแล้วที่หน้าร้าน |

**จำนวนที่ต้องจัดส่งจริง (effective quantity):**

- `quantity_to_deliver = quantity - quantity_picked_up_at_store`
- ใช้ค่าตัวนี้ทุกที่ที่เกี่ยวกับ **การจัดทริป / จัดเรียง / น้ำหนัก / พาเลท**

ข้อจำกัด:

- `quantity_picked_up_at_store >= 0`
- `quantity_picked_up_at_store <= quantity`

### 2. Sync จาก Order (trigger เดิม)

- ตอน sync จาก `order_items` → `delivery_trip_items`:
  - อัปเดต `quantity` = จำนวนจาก order (เหมือนเดิม)
  - ถ้า `quantity` ลดลง: ให้ cap `quantity_picked_up_at_store` ไม่ให้เกิน `quantity` ใหม่ (เช่น ถ้าเดิมเบิก 5 แล้ว order ลดเหลือ 3 หน่วย ให้ตั้งเบิก = 3 หรือ 0 ตามนโยบาย)
  - ถ้า `quantity` เพิ่มขึ้น: ไม่ต้องเปลี่ยน `quantity_picked_up_at_store` (เบิกไปแล้วคงเดิม)

---

## จุดในระบบที่ต้องใช้ "จำนวนที่ต้องจัดส่ง"

ใช้ **effective quantity** (= `quantity - quantity_picked_up_at_store`) ใน:

1. **หน้าจัดทริป / สร้างทริป** — รายการสินค้าต่อร้าน, น้ำหนัก, จำนวนพาเลท
2. **หน้าจำลองจัดเรียง (Packing Simulator)** — จำนวนที่เหลือให้จัด, สถิติ
3. **API / service** ที่คำนวณน้ำหนักทริป, ความจุรถ, bin packing
4. **รายงาน/สรุปทริป** — แสดงทั้ง "สั่ง X" และ "เบิกไป Y" และ "จัดส่ง Z" ได้ถ้าต้องการ

---

## UI ที่ต้องเพิ่ม/แก้

1. **รายการสินค้าต่อร้าน (ในทริป/ออเดอร์)**
   - แสดง: สั่ง **X** | เบิกที่ร้าน **Y** | เหลือจัดส่ง **Z** (หรือเฉพาะ Z ถ้า Y=0)
   - ช่องแก้ไข "เบิกที่ร้านแล้ว" (หรือปุ่ม "รับที่หน้าร้านทั้งหมด" ให้ Z=0)

2. **การตรวจสอบก่อนออกทริป**
   - แจ้งเตือนถ้ามีรายการที่ Z=0 (ไม่ต้องส่ง) ให้ตัดออกจากทริปหรือระบุว่า "รับที่ร้านแล้ว"

3. **(ถ้าต้องการ) ระดับ Order**
   - ถ้าต้องการบันทึกที่ออเดอร์ด้วย อาจเพิ่ม `quantity_picked_up_at_store` ที่ `order_items` แล้ว sync ลงทริป; หรือให้บันทึกเฉพาะที่ `delivery_trip_items` ก็ได้

---

## ขั้นตอนการทำ (สรุป)

1. **Migration SQL**  
   - เพิ่ม `quantity_picked_up_at_store` ใน `delivery_trip_items`  
   - CHECK constraint และ comment  
   - (ถ้าต้องการ) ปรับ trigger sync ให้ cap ค่าเบิกเมื่อ order quantity ลดลง

2. **Backend / API**  
   - คืนค่า `quantity_picked_up_at_store` และ/หรือ `quantity_to_deliver`  
   - ทุกจุดที่คำนวณจาก "จำนวนสินค้าในทริป" ให้ใช้ `quantity_to_deliver`

3. **Frontend**  
   - แสดงและแก้ไข "เบิกที่ร้านแล้ว"  
   - ใช้ `quantity_to_deliver` ใน Packing Simulator และหน้าจัดทริป

4. **ทดสอบ**  
   - สร้างออเดอร์ → เบิกส่วนหนึ่ง → ตรวจว่าใบจัดทริป/จัดเรียงใช้จำนวนที่ต้องส่งจริง

---

## หมายเหตุ

- **ใบแจ้งหนี้/ยอดขาย:** ใช้ `quantity` (จำนวนที่สั่ง) เป็นหลัก  
- **การจัดทริป/จัดเรียง/โหลดรถ:** ใช้ `quantity_to_deliver` เท่านั้น  
- ถ้าลูกค้ารับที่หน้าร้านทั้งหมด: ตั้ง `quantity_picked_up_at_store = quantity` → เหลือจัดส่ง 0 รายการนั้นไม่ต้องขึ้นรถ
