# การออกแบบให้ AI เรียนรู้จากข้อมูลการจัดเรียงสินค้าได้อย่างแม่นยำ

**เป้าหมาย:** ให้ AI แนะนำการจัดเรียงได้ตรงกับของจริง แม้การจัดเรียงจะมีหลายรูปแบบ (พาเลทหลายแบบ, ชั้นหลายชั้น, บนพื้น ฯลฯ)

---

## 1. ปัญหาที่ต้องแก้

- **การจัดเรียงมีหลายรูปแบบ:** พาเลท 1–N, แต่ละพาเลทมีหลายชั้น, มีโซน "บนพื้น", สินค้าหลากชนิด
- **AI ต้องเรียนรู้จากของจริง:** ข้อมูลใน `trip_packing_layout` + `trip_packing_layout_items` คือ "คำตอบที่ถูก" ที่คนจัดไปแล้ว
- **ความแม่นยำ:** ยิ่งมีประวัติ layout มากและส่งเข้า AI ได้เป็นระบบ ยิ่งแนะนำได้ตรง

---

## 2. แนวทางหลัก: ข้อมูล + วิธีเรียนรู้

### 2.1 ข้อมูลที่มีอยู่แล้ว (ใช้ได้ทันที)

| แหล่งข้อมูล | รายละเอียด | ใช้สำหรับ |
|-------------|------------|-----------|
| `trip_packing_layout` | แต่ละพาเลท/พื้น, position_index, total_layers | โครงสร้างตำแหน่ง |
| `trip_packing_layout_items` | สินค้าในแต่ละตำแหน่ง, quantity, layer_index | ว่าใส่อะไรที่ไหน |
| `exportDetailedTrainingData()` | ทริป + รถ + สินค้า + metrics | ส่งไปเทรน/วิเคราะห์ |
| `getPackingPatternInsights()` | น้ำหนักเฉลี่ยต่อพาเลทจากประวัติ | ส่งเป็นข้อความให้ Gemini |
| `getProductPackingProfiles()` | สินค้าแต่ละตัวเคยอยู่พาเลทไหน, กี่ชั้น | ส่งเป็นข้อความให้ Gemini |

เพื่อให้ AI เรียนรู้ได้**แม่นยำ** ต้องทำสามอย่างหลัก:

1. **ส่งประวัติ layout จริงเข้า AI เป็นข้อความ/โครง** (ทริปคล้ายกัน + สรุปการจัดเรียง)
2. **ลดความหลากหลายของ "รูปแบบ"** โดยการ normalize หรือจัดกลุ่ม pattern
3. **คำนวณที่ซับซ้อน (ถ้ามี)** ทำใน Python แล้วค่อยส่งผลกลับมา

---

## 3. ให้ AI เรียนรู้จากข้อมูลการจัดเรียงได้แม่นยำขึ้น

### 3.1 ส่ง "การจัดเรียงจริงจากทริปคล้ายกัน" เข้า Prompt (ทำได้ในระบบปัจจุบัน)

- ตอนนี้มี `getTripPackingLayout()` และ similar trips อยู่แล้ว
- **เพิ่มขั้นตอน:** สำหรับทริปคล้ายกันที่**มี layout บันทึก** ให้ดึง layout มาแล้วสรุปเป็นข้อความ เช่น  
  `พาเลท 1 (450kg): น้ำดื่ม 50 ลัง, ผงซักฟอก 20 แพ็ค; พาเลท 2 (400kg): ทิชชู่ 40 แพ็ค; บนพื้น: สบู่ 30 ชิ้น`
- ส่งข้อความนี้เข้า **historical_context** หรือ **packing_patterns** ใน Edge Function `ai-trip-recommendation`
- **ผลลัพธ์:** AI (Gemini) ได้เห็น "ตัวอย่างการจัดเรียงจริง" จึงแนะนำได้ใกล้เคียงของจริงมากขึ้น

**การ normalize รูปแบบหลายแบบ:**  
ไม่ต้องให้ AI จำทุกรูปแบบเอง แค่ส่งเป็น **ข้อความอธิบาย** (เช่น "พาเลทที่ 1 มีอะไรบ้าง ชั้นล่าง/บนมีอะไร") ให้เหมือนกันทุกทริป จะช่วยให้ AI เรียนรู้ pattern จากข้อความได้

### 3.2 จัดกลุ่ม Pattern (ลดความซับซ้อน)

การจัดเรียงมีหลายรูปแบบ แต่สามารถ**จัดกลุ่ม**ให้เป็นประเภทใหญ่ๆ ได้ เช่น:

- **ประเภท A:** ทริปน้ำหนักน้อย, 1–2 พาเลท, สินค้าไม่ fragile
- **ประเภท B:** ทริปน้ำหนักมาก, 4+ พาเลท, มีของหนัก+เบา
- **ประเภท C:** มีสินค้าคุมอุณหภูมิ, ใช้พื้นที่เฉพาะ

วิธีทำ:

1. **ใน TypeScript (หรือใน Python):** คำนวณ feature ของทริป เช่น  
   `จำนวนพาเลท, น้ำหนักรวม, จำนวนชนิดสินค้า, มีของเย็นหรือไม่, มีของ fragile หรือไม่`
2. **จับคู่ทริปใหม่กับทริปเก่า** ที่ feature ใกล้เคียงกัน แล้วเอา layout ของทริปเก่าที่ "คล้ายที่สุด" มาสรุปส่งให้ AI
3. **ผล:** AI ไม่ต้องเดาทั้งก้อน แค่ได้ "ตัวอย่างที่คล้ายกันมาก" มาปรับใช้

การจัดกลุ่ม/คลัสเตอร์ (เช่น K-means) หรือ similarity score ที่ซับซ้อน ถ้าต้องการทำดีมาก สามารถย้ายไปทำใน **Python** (ดูหัวข้อ 5)

### 3.3 ใช้ประวัติโดยตรงเป็น "ตัวอย่างแนะนำ" (Retrieval-based)

- **เก็บ:** ทุกทริปที่ completed และมี layout บันทึก → สรุปเป็น "layout summary" (ข้อความหรือ JSON สั้นๆ)
- **ทริปใหม่:** คำนวณความคล้ายกับทริปเก่า (จาก น้ำหนัก, จำนวนร้าน, ชุดสินค้า, รถ ฯลฯ)
- **ดึง:** ทริปเก่าที่คล้ายที่สุด 3–5 ทริป พร้อม layout summary
- **ส่งให้ AI:** "ทริปที่คล้ายกันเคยจัดแบบนี้: [สรุป 1], [สรุป 2], ... จงแนะนำการจัดเรียงสำหรับทริปใหม่นี้"
- **ผล:** AI เรียนรู้จากของจริงโดยไม่ต้องเทรนโมเดลใหม่ แค่ได้ตัวอย่างที่ดีและเกี่ยวข้อง

แนวทางนี้ทำได้ใน **TypeScript** ทั้งหมด (คำนวณ similarity แบบง่าย เช่น weighted score ของน้ำหนัก/จำนวนพาเลท/สินค้า) หรือถ้าต้องการ similarity ที่ซับซ้อน (เช่น embedding) ค่อยใช้ Python

## 4. เมื่อไหร่ต้องใช้ Python มาช่วยคำนวณ

### 4.1 ที่ TypeScript ทำได้ดีอยู่แล้ว

- ดึง layout จาก DB, สรุปเป็นข้อความ, ส่งเข้า Gemini
- คะแนนความคล้ายแบบง่าย (น้ำหนักใกล้เคียง, ชุดร้านเดียวกัน, จำนวนพาเลทใกล้เคียง)
- Rule-based bin packing แบบง่าย (เช่น ใน `utils/palletPacking.ts`)

### 4.2 ที่ Python จะช่วยได้ชัดเจน

| งาน | เหตุผล | ตัวอย่างเครื่องมือ |
|-----|--------|-------------------|
| **Bin packing / 3D packing ที่ซับซ้อน** | ปัญหา NP-hard, มีอัลกอริทึมและไลบรารีสำเร็จรูปใน Python | OR-Tools, PuLP, custom heuristics |
| **การเทรนโมเดลจากประวัติ layout** | ใช้ ML framework ได้สะดวก | scikit-learn, XGBoost, PyTorch (ถ้าต้อง embedding) |
| **Clustering / จัดกลุ่ม pattern** | มีฟังก์ชันสำเร็จรูป | scikit-learn (KMeans, DBSCAN) |
| **Embedding สำหรับความคล้าย** | สร้าง vector จาก "layout summary" แล้วหาทริปคล้าย | sentence-transformers, numpy |
| **การคำนวณตัวเลขหนัก (สถิติ, การปรับโมเดล)** | ทำใน Python แล้วส่งผลกลับเป็น API | FastAPI + โมเดลที่เทรนไว้ |

สรุป: **ไม่จำเป็นต้องใช้ Python ทุกอย่าง** แต่ถ้าต้องการ  
(1) การจัดเรียงที่ optimal มากขึ้น (bin packing แบบซับซ้อน) หรือ  
(2) เรียนรู้จากข้อมูล layout แบบอัตโนมัติ (เทรนโมเดล / embedding / clustering)  
การใช้ **Python แยกเป็น service เล็กๆ** แล้วให้ Frontend หรือ Edge Function เรียกผ่าน HTTP จะเหมาะสม

## 5. สถาปัตยกรรมที่แนะนำ (แบบใช้ Python เมื่อจำเป็น)

```
┌─────────────────────────────────────────────────────────────────┐
│  Vehicle Control Center (TypeScript / Supabase)                  │
│  - บันทึก layout จริง (trip_packing_layout)                      │
│  - ดึง similar trips + layout summary                            │
│  - เรียก AI (Gemini) สำหรับข้อความแนะนำจัดเรียง                 │
│  - เรียก Python Service เมื่อต้องการคำนวณซับซ้อน                  │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTP (optional)
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  Python Service (FastAPI / Flask)                                 │
│  - รับ: ทริปใหม่ (สินค้า, น้ำหนัก, ขนาดรถ, จำนวนพาเลท)           │
│  - ดึงประวัติ: เรียก API ของเรา (exportDetailedTrainingData     │
│    หรือ GET /api/training-data?from=&to=) หรืออ่านจาก DB โดยตรง │
│  - งานที่ทำ:                                                     │
│    a) Bin packing ที่ซับซ้อน (OR-Tools / heuristic)              │
│    b) คำนวณ similarity จาก layout (embedding หรือ feature)      │
│    c) เทรนโมเดลเล็ก (เช่น XGBoost: input = trip features,       │
│       output = จำนวนพาเลทหรือ pattern id) แล้ว inference        │
│  - คืน: แผนจัดเรียง (JSON) หรือ pattern ที่แนะนำ                 │
└─────────────────────────────────────────────────────────────────┘
```

- **ขั้นต่ำ (ไม่ใช้ Python):**  
  - ส่ง layout summary ของทริปคล้ายกันเข้า Gemini  
  - ใช้ rule-based + similarity แบบง่ายใน TypeScript  
  → AI ก็เรียนรู้จากข้อมูลการจัดเรียงได้ดีขึ้นแล้ว  

- **ขั้นกลาง (ใช้ Python บางส่วน):**  
  - ใช้ Python ทำ bin packing ที่ซับซ้อน หรือทำ clustering ของ pattern  
  - คืนผลเป็น "แผนจัดเรียงที่แนะนำ" หรือ "pattern id"  
  - ฝั่งเราเอาไปแสดงหรือส่งต่อให้ Gemini เป็นข้อความอธิบาย  

- **ขั้นสูง (เทรนโมเดล):**  
  - Python เทรนโมเดลจาก `exportDetailedTrainingData` + layout จริง  
  - Inference รับทริปใหม่ → คืน "รูปแบบจัดเรียงที่แนะนำ" หรือคะแนนความคล้าย  
  - ระบบเราใช้ผลนี้เป็น input เพิ่มให้ Gemini หรือแสดงโดยตรง

## 6. แผนดำเนินการแบบเป็นขั้น (ไม่บังคับใช้ Python ทันที)

### Phase 1: ใช้ข้อมูลที่มีให้ AI เรียนรู้ได้ดีขึ้น (ทำใน TypeScript + Edge Function)

1. **รวม layout summary ใน similar-trips context**
   - ใน `buildSimilarTripsContext` หรือที่ที่ส่ง `historical_context` ให้ AI  
   - สำหรับทริปคล้ายกันที่มี layout: เรียก `getTripPackingLayout(tripId)` แล้วสรุปเป็นข้อความ (พาเลทที่ 1 มีอะไร, พาเลทที่ 2 มีอะไร, บนพื้นมีอะไร)
   - ใส่ข้อความนี้ลงใน `historical_context` / `packing_patterns`

2. **ส่ง layout summary เข้า Edge Function**
   - ใน `ai-trip-recommendation`: รับ `historical_context` ที่มีทั้งข้อความทริปคล้ายและ**สรุปการจัดเรียงจริง** แล้วใส่ใน prompt
   - ปรับ prompt ให้บอกชัดว่า "ใช้ตัวอย่างการจัดเรียงจริงด้านล่างเป็นแนวทาง"

3. **ทดสอบกับทริปจริง**
   - เลือกทริปที่มี layout บันทึกแล้ว สร้างทริปใหม่ที่คล้ายกัน แล้วดูว่า AI แนะนำจัดเรียงใกล้เคียงของจริงหรือไม่

ผลที่ได้: **AI เรียนรู้จากข้อมูลการจัดเรียงได้อย่างแม่นยำขึ้นโดยไม่ต้องใช้ Python** เพราะได้เห็นตัวอย่างการจัดเรียงจริงจากทริปที่คล้ายกัน

### Phase 2 (ถ้าต้องการความแม่นยำสูงขึ้น): Normalize + จัดกลุ่ม pattern

1. **กำหนดรูปแบบสรุป layout ให้คงที่**
   - ทุกทริปสรุปเป็นรูปแบบเดียวกัน เช่น  
     `[พาเลท|พื้น](index): [สินค้า1 qty1, สินค้า2 qty2, ...]; ...`
   - จะได้ข้อความที่ AI เปรียบเทียบและเรียนรู้ pattern ได้ง่าย

2. **คำนวณ similarity ของทริปใหม่กับทริปเก่า**
   - ใช้ feature: น้ำหนักรวม, จำนวนพาเลท, จำนวนชนิดสินค้า, มีของเย็น/ของหนักหรือไม่
   - เลือกทริปเก่าที่คะแนนสูงสุด 3–5 ทริป แล้วส่งเฉพาะ layout summary ของทริปเหล่านี้ให้ AI  
   - ทำได้ใน TypeScript; ถ้าอยากใช้ clustering หรือ metric ที่ซับซ้อน ค่อยย้ายไป Python

### Phase 3 (ถ้าต้องการการจัดเรียงที่ optimal จริง): ใช้ Python

1. **สร้าง Python service เล็ก**
   - รับ input: สินค้า, น้ำหนัก, ขนาดรถ, จำนวนพาเลท (และ constraint อื่นๆ ถ้ามี)
   - ใช้ OR-Tools หรือ heuristic ทำ bin packing / 3D packing
   - คืนผลเป็น JSON (ตำแหน่งพาเลท/ชั้น กับสินค้า)

2. **เชื่อมกับระบบเรา**
   - Edge Function หรือ Backend เรียก Python service แล้วนำผลที่ได้ไปแสดงหรือส่งให้ Gemini เป็น "แผนที่แนะนำ" เพื่อให้ AI อธิบายหรือปรับรายละเอียด

3. **(ถ้าต้องการ) เทรนโมเดลจากประวัติ**
   - Export ข้อมูลจาก `exportDetailedTrainingData` + layout จริง
   - เทรนใน Python (เช่น โมเดลทำนายจำนวนพาเลท หรือ pattern id)
   - เปิด API inference ให้ระบบเราเรียกใช้

## 7. สรุปคำตอบ

- **AI จะเรียนรู้จากข้อมูลการจัดเรียงได้อย่างแม่นยำขึ้นได้อย่างไร**  
  - ส่ง **การจัดเรียงจริงจากทริปที่คล้ายกัน** (layout summary) เข้า prompt ให้ Gemini  
  - **ลดความหลากหลายของรูปแบบ** โดยสรุป layout แบบเดียวกันทุกทริป และถ้าต้องการจัดกลุ่ม pattern (คล้ายกันมาก่อน แล้วค่อยส่งเฉพาะตัวอย่างที่เกี่ยวข้อง)  
  - ถ้าต้องการความแม่นยำสูงมาก: ใช้ประวัติเป็น training data เทรนโมเดลหรือทำ embedding ใน Python แล้วส่งผลกลับมา

- **ต้องใช้ Python หรือไม่**  
  - **ไม่จำเป็น** สำหรับการให้ AI เรียนรู้จากข้อมูลการจัดเรียงในระดับที่ "ได้ตัวอย่างการจัดเรียงจริงจากทริปคล้ายกัน" และ "แนะนำเป็นข้อความได้ดีขึ้น" — ทำได้ใน TypeScript + Gemini  
  - **ควรใช้ Python** เมื่อต้องการ (1) การคำนวณคณิตศาสตร์/อัลกอริทึมที่ซับซ้อน (bin packing, optimization, clustering) หรือ (2) การเทรนโมเดล/embedding จากประวัติ layout เพื่อความแม่นยำสูงสุด

ถ้าต้องการเริ่มจากจุดที่ลงมือทำได้ทันที แนะนำเริ่มจาก **Phase 1**: รวม layout summary ของทริปคล้ายกันเข้า `historical_context` / prompt ของ AI แล้วทดสอบกับทริปจริงก่อน จากนั้นค่อยตัดสินใจทำ Phase 2 (จัดกลุ่ม pattern) หรือ Phase 3 (Python) ตามความต้องการความแม่นยำและทรัพยากรที่มี
