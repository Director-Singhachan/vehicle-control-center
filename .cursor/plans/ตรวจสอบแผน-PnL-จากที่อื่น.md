# การตรวจสอบแผน "ระบบจัดการรายได้ ต้นทุน และวิเคราะห์กำไร (Trip, Vehicle, Fleet P&L)"

> **เอกสารอ้างอิง/ประวัติการตรวจสอบ** — ไม่ใช้เป็นแผนดำเนินการ  
> แผนที่ใช้ดำเนินการจริงคือ **ต้นทุนคงที่-ผันแปร_และตัวชี้ต่อกม._เที่ยว_ชิ้น_1f277858.plan.md** (ฉบับรวม: รายได้/ต้นทุน/กำไร 3 ระดับ + Daily Rate + Idle + Utilization + ตัวชี้ต่อกม./เที่ยว/ชิ้น)

**จุดมุ่งหมายของคุณ:** ผู้มองเห็นทุกรายละเอียดของผลประกอบการ (จากจุลภาครายเที่ยว → รถ 1 คัน → มหภาคทั้งกอง)

---

## 1. สรุป: แผนที่เสนอสอดคล้องกับจุดมุ่งหมายหรือไม่

**สอดคล้องดี** — โครง 3 ระดับ (Trip P&L → Vehicle P&L → Fleet P&L) ทำให้เห็นผลประกอบการได้ครบตามที่ต้องการ:

| ระดับ | สิ่งที่เห็น | สอดคล้องกับ "ทุกรายละเอียด" |
|-------|-------------|-----------------------------|
| **Trip P&L** | กำไร/ขาดทุนต่อเที่ยว (รายได้เที่ยว − ต้นทุนผันแปรเที่ยว − ต้นทุนคงที่ตามวันวิ่ง) | เห็นว่างานไหนคุ้ม/ไม่คุ้ม ระดับจุลภาค |
| **Vehicle P&L** | กำไร/ขาดทุนรวมของรถ 1 คัน + **ต้นทุนจอดทิ้ง (Idle Cost)** | เห็นว่ารถคันไหนทำเงิน/เป็นภาระ และต้นทุนวันว่าง |
| **Fleet P&L** | กำไร/ขาดทุนสุทธิทั้งบริษัทตามช่วงเวลา | ภาพรวมผู้บริหาร |

ตัวชี้ **อัตราการใช้รถ (Utilization)** และ **ต้นทุนคงที่ต่อวัน (Daily Rate)** ช่วยให้ปันส่วนต้นทุนคงที่ลงระดับวัน/เที่ยวได้ชัด และเห็น "ต้นทุนจอดทิ้ง" ชัดเจน

## 2. เทียบกับแผนเดิมใน Repo และระบบปัจจุบัน

### 2.1 สิ่งที่แผนใหม่ **เสริม/ขยาย** จากแผนเดิม

| หัวข้อ | แผนเดิม (ใน repo) | แผนที่เสนอใหม่ |
|--------|-------------------|-----------------|
| **รายได้ (Revenue)** | ไม่ได้ระบุที่เก็บ — ตัวชี้กำไร/Break-even "รอแหล่งข้อมูล" | เพิ่ม **trip_revenue** (และ trip_start_date, trip_end_date) ในเที่ยววิ่ง → รายได้ชัดระดับเที่ยว |
| **ต้นทุนคงที่** | ปันส่วนเป็น "ต้นทุนคงที่รวมในช่วง" แล้วหารด้วย กม./เที่ยว/ชิ้น | แปลงเป็น **Daily Fixed Cost** ก่อน แล้วคูณจำนวนวัน (ของเที่ยว หรือของช่วง) → ใช้คำนวณ Trip P&L และ Idle Cost ได้ตรง |
| **ระดับรายงาน** | เน้นรายงานการใช้รถ + ต้นทุนต่อกม./เที่ยว/ชิ้น | เพิ่ม **3 ระดับ**: Trip P&L, Vehicle P&L, Fleet P&L + Dashboard มหภาค |
| **ต้นทุนจอดทิ้ง** | ไม่มี | มี **Idle Cost** (ต้นทุนคงที่ของวันที่ไม่ได้วิ่ง) |
| **Utilization** | ไม่มี | **อัตราการใช้รถ** = จำนวนวันที่วิ่งงาน ÷ จำนวนวันในช่วง |

### 2.2 สิ่งที่ **ตรงกัน** กับแผนเดิมและระบบ

- ตาราง **vehicle_fixed_costs** และ **vehicle_variable_costs** — โครงสร้างใกล้เคียงกัน (แผนเดิมมี period_type, แผนใหม่มี period_start/period_end; แผนเดิม delivery_trip_id เป็น nullable)
- ต้นทุนผันแปรจาก **fuel_records, commission_logs, tickets/ticket_costs** — ยังใช้ร่วมได้
- UI บันทึกต้นทุนที่ **หน้ารถ + โมดัล** — สอดคล้องกับแนวทางที่เลือกไว้แล้ว

### 2.3 สิ่งที่ **ต้องตรวจ/ตัดสินใจ** เพราะต่างจากแผนเดิมหรือระบบ

| หัวข้อ | รายละเอียด | คำแนะนำ |
|--------|-------------|----------|
| **รายได้ระดับเที่ยว** | ตอนนี้ระบบคำนวณรายได้จาก **orders.total_amount** ที่ผูก delivery_trip_id (ใน vehicleTripUsageService) — ไม่มีฟิลด์ trip_revenue ใน delivery_trips | **ทางเลือก:** (ก) เพิ่มคอลัมน์ **trip_revenue** (และ trip_start_date, trip_end_date) ใน delivery_trips ตามแผนใหม่ เพื่อให้กรอก/แก้ไขได้ตรงระดับเที่ยว และใช้คำนวณ Trip P&L ได้ทันที (ข) ไม่เพิ่ม แล้วดึงรายได้จาก sum(orders) ต่อเที่ยวเหมือนเดิม — ได้รายได้ระดับเที่ยวแต่ต้องมีออเดอร์ผูกเที่ยว และต้องมีวิธีหา "จำนวนวันของเที่ยว" จากที่อื่น (เช่น trip_log checkout–checkin) |
| **trip_start_date / trip_end_date** | delivery_trips ปัจจุบันมีแค่ **planned_date** | เพื่อคำนวณ "ต้นทุนคงที่ × จำนวนวันของเที่ยว" ต้องมี "ระยะเวลาของเที่ยว" — แผนใหม่เสนอ trip_start_date + trip_end_date。ทางเลือก: หาจาก trip_log (checkout → checkin) แทนได้ ถ้ามีข้อมูลครบ |
| **delivery_trip_id ใน vehicle_variable_costs** | แผนเดิม: **nullable** (ผูกเที่ยวได้ถ้ามี)。แผนใหม่: "ผูก ID เที่ยวเสมอถ้าเกิดตอนวิ่งงาน, ปล่อยว่างได้ถ้าเป็นค่าซ่อมบำรุงทั่วไป" | ความหมายเหมือนกัน — ใช้ **nullable** ได้；ใน UI บังคับเลือกเที่ยวเมื่อ cost_type เกี่ยวกับเที่ยว (ทางด่วน, เบี้ยเลี้ยง ฯลฯ) และปล่อยว่างเมื่อเป็นซ่อมทั่วไป |
| **จำนวน Phases** | แผนเดิม 5 Phase (Schema, ภาษี/ประกัน, UI ต้นทุน, ผูกรายงาน, Export)。แผนใหม่ 6 Phase แล้วแยก Trip/Vehicle/Fleet report ชัด | รวมได้: Phase 1 = Schema + Service Daily Fixed Cost; Phase 2 = UI รายได้+ต้นทุน; Phase 3 = Trip P&L; Phase 4 = Vehicle P&L (+ Idle); Phase 5 = Fleet Dashboard; Phase 6 = Export/รายเดือน (optional) |

## 3. ความสมบูรณ์ของสูตรและ Logic

- **ต้นทุนคงที่ต่อวัน** = ต้นทุนคงที่รวมในรอบ ÷ จำนวนวันในรอบ — ถูกต้อง
- **กำไรสุทธิต่อเที่ยว** = รายได้เที่ยว − (ผันแปรของเที่ยว + ต้นทุนคงที่ต่อวัน × จำนวนวันของเที่ยว) — ถูกต้อง
- **กำไรสุทธิรถ 1 คัน** = รายได้รวมของรถ − (ผันแปรทั้งหมดในช่วง + ต้นทุนคงที่ต่อวัน × จำนวนวันใน Filter) — ถูกต้อง；จำนวนวันใน Filter คือ "จำนวนวันทั้งหมดในช่วง" จึงได้ **Idle Cost** = ต้นทุนคงที่ต่อวัน × วันที่ไม่ได้วิ่ง
- **Fleet P&L** = SUM(กำไรสุทธิของรถทุกคัน) — ถูกต้อง
- **Utilization** = จำนวนวันที่วิ่งงาน ÷ จำนวนวันทั้งหมด — ถูกต้อง

ไม่มีสูตรในแผนที่ขัดกับหลักการบัญชีต้นทุนหรือ P&L。

## 4. สิ่งที่ควรเพิ่มหรือระบุให้ชัดในแผน

เพื่อให้ "มองเห็นทุกรายละเอียด" และทำได้จริงในระบบนี้:

1. **ที่มาของ "จำนวนวันของเที่ยว"**  
   - ถ้าใช้ trip_start_date / trip_end_date — ต้องเพิ่มใน Schema และ UI จัดการเที่ยว。  
   - ถ้าใช้จาก trip_log (checkout → checkin) — ระบุในแผนว่า "ระยะเวลาเที่ยว = จำนวนวันระหว่าง checkout ถึง checkin (หรือถึงวันนี้ถ้ายังไม่ checkin)"

2. **ที่มาของ "จำนวนวันที่วิ่งงาน" (สำหรับ Utilization และ Idle)**  
   - กำหนดนิยามชัด: นับจากวันที่มีเที่ยว completed หรือมี trip_log checkout ในวันนั้น (และกรอง vehicle_id ตามรถที่เลือก)

3. **รายได้ระดับเที่ยว (สำหรับ Trip P&L)**  
   - ถ้าใช้ trip_revenue: ระบุว่าบันทึกที่ไหน (หน้าจัดการเที่ยววิ่ง / ฟอร์มแก้เที่ยว) และว่าคำนวณจากออเดอร์อัตโนมัติได้หรือไม่ (เช่น default = sum(orders ที่ผูกเที่ยว))  
   - ถ้าไม่ใช้ trip_revenue: ระบุว่า Trip P&L ใช้รายได้จาก sum(orders.total_amount where delivery_trip_id = เที่ยวนี้)

4. **การรวมต้นทุนเดิม (น้ำมัน, ค่าคอม, ค่าซ่อม)**  
   - ระบุว่าใน Trip P&L / Vehicle P&L จะรวม fuel_records, commission_logs, ticket_costs กับ vehicle_variable_costs อย่างไร (เช่น ผันแปรต่อเที่ยว = variable_costs ที่ delivery_trip_id ตรง + สัดส่วนน้ำมัน/ค่าคอมของเที่ยวนั้น ถ้ามีการปันส่วน)

5. **RLS และสิทธิ์**  
   - vehicle_fixed_costs / vehicle_variable_costs ใช้ RLS ตาม vehicle/branch เหมือนฟีเจอร์อื่น — ระบุใน Phase 1 เพื่อไม่ลืม

## 5. สรุปและข้อเสนอ

- แผนที่เสนอ **สอดคล้องกับจุดมุ่งหมาย** "ผู้มองเห็นทุกรายละเอียดของผลประกอบการ" และ **น่าจะมีประสิทธิภาพ** เพราะมีโครง 3 ระดับชัด (Trip → Vehicle → Fleet) และใช้ Daily Fixed Cost + Idle Cost + Utilization ได้ตรงกับความต้องการบริหารขนส่ง。
- **แนะนำให้รับแนวทางนี้เป็นหลัก** และนำไปรวมกับแผนเดิมใน repo โดย:
  - ใช้ **ลำดับ Phase ตามแผนใหม่** (Schema + Daily Rate Service → UI รายได้/ต้นทุน → Trip P&L → Vehicle P&L → Fleet Dashboard)
  - **เก็บตัวชี้เดิม** (ต้นทุนต่อกม./ต่อเที่ยว/ต่อชิ้น) ไว้ในรายงานระดับรถหรือรายงานการใช้รถได้ เพราะตอบคำถามคนละแบบ
  - **ตัดสินใจที่มาของรายได้และจำนวนวันของเที่ยว** ตามข้อ 4 แล้วเขียนลงแผนให้ชัด (trip_revenue + trip_start/end หรือ derive จาก orders + trip_log)

ถ้าต้องการ ผมสามารถช่วย **รวมแผนใหม่เข้ากับไฟล์แผนเดิม** (ต้นทุนคงที่-ผันแปร_และตัวชี้ต่อกม._เที่ยว_ชิ้น) เป็นฉบับเดียวได้ เช่น เพิ่มส่วน "PnL 3 ระดับ + Daily Rate + Idle + Utilization" และปรับลำดับ Phase ให้สอดคล้องกับแผนที่เสนอใหม่。
